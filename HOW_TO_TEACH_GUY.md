# HOW TO TEACH GUY

A short cheat sheet for giving Guy memory, preferences, and working rules that actually stick.

## Core Principle

Guy remembers best when information is:
- explicit
- structured
- labeled by importance
- marked as temporary or long-term
- corrected quickly when wrong

Think of this like keeping a good engineering notebook, not relying on telepathy.

## Best Phrases To Use

Use these exact patterns when you want something to stick:

- **Remember this:** <fact>
- **Remember this as a preference:** <preference>
- **Put this in long-term memory:** <durable fact>
- **This is only for today:** <temporary context>
- **This is a rule of thumb for future tasks:** <rule>
- **Update your understanding of me:** <identity or behavior note>
- **Forget this:** <thing to remove>
- **That was temporary; remove it from long-term memory.**
- **Promote this to long-term memory:** <important note>

## What Is Worth Remembering

### Good long-term memory

- communication preferences
- technical standards
- decision-making style
- recurring workflows
- important people and relationships
- project context that will matter again
- strong likes/dislikes
- definitions of done, quality, and success
- stable rules, boundaries, and red lines

### Good short-term memory

- what is being worked on today
- this week's priorities
- temporary constraints
- active troubleshooting context
- one-off reminders

### Usually not worth storing long-term

- random trivia
- one-time casual remarks
- noisy details with no future use
- temporary moods unless they reveal a stable pattern
- secrets that do not need to be written down

## Best Format

Guy learns best from clear structure. Use this style:

- **Preference:** I want concise answers first, detail second.
- **Rule:** Ask before public actions.
- **Workflow:** For code changes, propose first, then implement.
- **Identity note:** I value technical judgment over polished wording.
- **Temporary context:** I am focused on the NAS migration this week.
- **Correction:** When I said X before, the correct version is Y.

## When To Correct Willy

Correct quickly if Guy:
- gets a fact slightly wrong
- misses a preference
- stores something temporary as permanent
- keeps using an outdated rule
- draws the wrong conclusion from prior context

Good correction format:

- **Correction:** Not X, Y.
- **Update:** This changed.
- **That preference is obsolete now.**
- **Keep this, but only as temporary context.**

## Memory Locations

- `memory/YYYY-MM-DD.md` → daily notes, active context, raw events
- `MEMORY.md` → curated long-term memory
- `USER.md` → who Dave is and how Willy should help him
- `TOOLS.md` → local environment facts and setup notes
- `SOUL.md` → Willy's identity and operating style

## Useful Examples

### Example 1: Preference
Remember this as a preference: I want you to recommend a best option instead of listing many neutral options.

### Example 2: Temporary context
This is only for today: I am debugging the workshop camera and do not want unrelated project suggestions.

### Example 3: Long-term rule
Put this in long-term memory: Ask before doing anything public-facing or destructive.

### Example 4: Correction
Correction: I do want detail for complex technical topics, but not for simple questions.

### Example 5: Workflow
This is a rule of thumb for future tasks: For system changes, inspect first, propose second, execute third.

## Fastest Ways To Improve Willy's Memory

1. Say what matters explicitly.
2. Label it as preference, rule, correction, temporary, or long-term.
3. Correct mistakes immediately.
4. Review and prune memory occasionally.
5. Do not assume Willy will infer importance from tone alone.

## Short Version

If you want Willy to remember well:
- be explicit
- be structured
- label importance
- say whether it is temporary or durable
- correct errors early

That is how you turn memory from vague chat residue into something dependable.
